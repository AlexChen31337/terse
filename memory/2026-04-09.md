## CC CLI OAuth Investigation (Late night Apr 8 → early Apr 9)

### Key Findings
- **Claude Code CLI OAuth flow:** Token from `~/.claude/.credentials.json` → `claudeAiOauth.accessToken` (or `CLAUDE_CODE_OAUTH_TOKEN` env var). SDK uses `authToken` (not `apiKey`), sends `Authorization: Bearer` + `anthropic-beta: oauth-2025-04-20`. Refresh via `https://platform.claude.com/v1/oauth/token`.
- **CC CLI plugin uses proxy-0 token** from OpenClaw config (`~/.openclaw/openclaw.json` → `models.providers.anthropic-proxy-0.apiKey`), NOT Claude Code's personal OAuth token. So CC CLI plugin is NOT consuming Claude Code's "extra usage" quota.
- **Direct API tests with proxy-0 token:** Haiku ✅, Opus/Sonnet rate-limited (quota exhausted). proxy-1 token: all rate-limited.
- **CC CLI plugin still works for Opus 4.6** even with the old code (Apr 7 version) — Claude CLI handles OAuth internally.

### Decisions
- **REVERTED commit `059c633`** on `AlexChen31337/openclaw-plugin-claude-code` — the Bearer auth fix was unnecessary because CC CLI handles OAuth tokens internally. Repo back to `50561c3`.
- OAuth Bearer fix approach was technically correct for direct API calls but not needed for the CC CLI plugin which spawns the `claude` CLI process.
