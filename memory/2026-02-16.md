## EvoClaw General-Purpose Scheduler Implementation

**Critical gap identified by Bowen:** "Without cron or heartbeat.md, would it be a disadvantage for evoclaw? In a use case that evoclaw is installed in an IOT edge device, which has to read the sensor data, analysis and write the report via a particular time interval?"

**The problem:**
- EvoClaw had periodic tasks ONLY for memory consolidation (hardcoded tickers)
- No general-purpose scheduling for IoT use cases
- Edge devices need self-contained periodic tasks (no external cron/systemd)
- Real scenario: Pi reads sensor every 5 min, analyzes every hour, syncs to cloud every 6 hours

**Solution implemented:**
Built complete scheduler system from scratch in ~4 hours.

**Architecture:**
```
internal/scheduler/
  ├── scheduler.go (coordinator, 269 lines)
  ├── job.go (definitions + validation, 209 lines)
  └── runner.go (execution engine, 205 lines)
```

**Configuration (evoclaw.json):**
```json
{
  "scheduler": {
    "enabled": true,
    "jobs": [
      {
        "id": "sensor-read",
        "name": "Read Temperature Sensor",
        "enabled": true,
        "schedule": {"kind": "interval", "intervalMs": 300000},
        "action": {"kind": "shell", "command": "./sensors/read_temp.sh"}
      },
      {
        "id": "hourly-analysis",
        "schedule": {"kind": "interval", "intervalMs": 3600000},
        "action": {
          "kind": "agent",
          "agentId": "iot-agent",
          "message": "Analyze sensor data and report anomalies"
        }
      },
      {
        "id": "cloud-sync",
        "schedule": {"kind": "cron", "expr": "0 */6 * * *"},
        "action": {
          "kind": "mqtt",
          "topic": "iot/sync",
          "payload": {"device": "pi-001", "type": "full-sync"}
        }
      },
      {
        "id": "daily-report",
        "schedule": {"kind": "at", "time": "09:00", "timezone": "America/New_York"},
        "action": {
          "kind": "http",
          "method": "POST",
          "url": "https://api.example.com/reports",
          "headers": {"Authorization": "Bearer TOKEN"},
          "payload": {"device": "pi-001"}
        }
      }
    ]
  }
}
```

**Schedule types:**
1. **interval** - Every N milliseconds (e.g., every 5 minutes)
2. **cron** - Standard cron expressions (0 */6 * * *)
3. **at** - Daily at specific time (HH:MM + timezone)

**Action types:**
1. **shell** - Run scripts/commands
2. **agent** - Send message to agent for AI processing
3. **mqtt** - Publish to MQTT topic
4. **http** - Make HTTP requests (POST/GET with headers + payload)

**CLI commands:**
```bash
evoclaw schedule list                  # List all jobs
evoclaw schedule add --config job.json # Add new job
evoclaw schedule remove sensor-read    # Remove job
evoclaw schedule status sensor-read    # Check status
```

**Integration:**
- Orchestrator implements `scheduler.Executor` interface:
  - `ExecuteAgent(ctx, agentID, message)` - sends message to agent inbox
  - `PublishMQTT(ctx, topic, payload)` - publishes via MQTT channel
- Starts/stops with orchestrator lifecycle
- Each job runs in dedicated goroutine with ticker
- Automatic state tracking (runs, errors, duration)

**Example IoT workflow:**
1. Every 5 min: Shell script reads temperature sensor → writes JSON
2. Every hour: Agent reads JSON → analyzes trends → detects anomalies
3. Every 6 hours: MQTT publishes to cloud sync topic
4. Daily 9 AM: HTTP POST sends report to API

## Scheduler API Endpoints + Integration Patterns

### 1. Scheduler REST API (`internal/api/scheduler.go` - 265 lines)

**Endpoints:**
```
GET    /api/scheduler/status          - Scheduler stats
GET    /api/scheduler/jobs            - List all jobs
GET    /api/scheduler/jobs/:id        - Job details + runtime stats
POST   /api/scheduler/jobs/:id/run    - Trigger job immediately
PATCH  /api/scheduler/jobs/:id        - Update job (enable/disable)
POST   /api/scheduler/jobs            - Add new job at runtime
DELETE /api/scheduler/jobs/:id        - Remove job
```

**Features:**
- All operations return JSON
- Runtime management (no restart needed)
- State tracking (runs, errors, duration, next run time)
- Enable/disable jobs without removal
- Add jobs dynamically via API

**Example API usage:**
```bash
# Get scheduler status
curl http://localhost:8420/api/scheduler/status

# List jobs with runtime stats
curl http://localhost:8420/api/scheduler/jobs

# Trigger job immediately (bypass schedule)
curl -X POST http://localhost:8420/api/scheduler/jobs/sensor-read/run

# Disable job
curl -X PATCH http://localhost:8420/api/scheduler/jobs/sensor-read \
  -H "Content-Type: application/json" \
  -d '{"enabled": false}'

# Add new job
curl -X POST http://localhost:8420/api/scheduler/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "id": "new-job",
    "schedule": {"kind": "interval", "intervalMs": 600000},
    "action": {"kind": "shell", "command": "./task.sh"}
  }'
```

### 2. Integration Patterns Documentation (`docs/SCHEDULER-PATTERNS.md` - 532 lines)

**8 Common Patterns:**

**Pattern 1: Memory Consolidation**
- Replace hardcoded tickers with scheduler jobs
- 3 jobs: warm eviction (hourly), tree prune (daily), full consolidation (monthly)
- User-configurable intervals (no code changes)

**Pattern 2: IoT Sensor Reading**
- Every 5 min: Shell script reads sensor → writes JSON
- Every hour: Agent analyzes data → alerts on anomalies

**Pattern 3: Cloud Sync**
- Every 6 hours: MQTT publish triggers cloud backup
- Demonstrates MQTT action type

**Pattern 4: Daily Reports**
- Daily 9 AM: HTTP POST sends report to API
- Timezone-aware scheduling (America/New_York)

**Pattern 5: Health Checks**
- Every 15 min: Agent checks disk/memory/network
- Alerts if: disk >90%, memory >90%, network down

**Pattern 6: Backup Rotation**
- Daily 2 AM: Create backup
- Weekly Sunday 3 AM: Rotate old backups (keep 4 weeks)

**Pattern 7: Heartbeat/Ping**
- Every minute: HTTP ping to monitoring service
- Service alerts if missing >2 minutes

**Pattern 8: Model Performance Tracking**
- Hourly: Query API, log metrics to JSONL
- Feeds evolution engine

**Best Practices:**
- Avoid overlapping executions (use safe buffer times)
- Use appropriate action types (shell/agent/mqtt/http)
- Handle failures gracefully (fallback scripts)
- Log job outputs for debugging
- Timezone-aware schedules
- Monitor job state via API
- Test jobs manually before scheduling

### 2. Governance Integration (Architectural Alignment)

**Problem:** Governance protocols (WAL, VBR, ADL, VFM) exist in `internal/governance/` but were NOT integrated into orchestrator or new components (scheduler, memory CLI, router CLI).

**Per SELF-GOVERNANCE.md:**
> "Self-governance enables autonomous agents to maintain reliability, consistency, and cost-effectiveness without human supervision."

**Solution implemented:**

**internal/orchestrator/orchestrator.go:**
```go
// Added to struct
governance *governance.Manager

// Added import
"github.com/clawinfra/evoclaw/internal/governance"
"path/filepath"

// Added initialization (between memory and scheduler)
func (o *Orchestrator) initGovernance() error {
    govCfg := governance.DefaultConfig()
    govCfg.Logger = o.logger
    govCfg.BaseDir = filepath.Join(o.cfg.Server.DataDir, "governance")
    
    mgr, err := governance.NewManager(govCfg)
    if err != nil {
        return fmt.Errorf("create governance manager: %w", err)
    }
    
    o.governance = mgr
    o.logger.Info("governance protocols initialized", "base_dir", govCfg.BaseDir)
    return nil
}

// Added accessor
func (o *Orchestrator) GetGovernance() *governance.Manager {
    return o.governance
}
```

**Initialization order:**
1. Cloud sync (optional)
2. Memory system (optional)
3. **Governance protocols** ← NEW
4. Scheduler (optional)
5. Health registry

**Governance storage:**
```
~/.evoclaw/data/governance/
  ├── wal/      # Write-Ahead Log
  ├── vbr/      # Verify Before Reporting
  ├── adl/      # Anti-Divergence Limit
  └── vfm/      # Value-For-Money
```

**4 Protocols Now Available:**

**WAL (Write-Ahead Log):**
- Prevents context loss through session compaction
- Tracks: corrections, decisions, analysis, state changes
- Usage: `governance.WAL.Append(agentID, "correction", text)`

**VBR (Verify Before Reporting):**
- Prevents false completion claims
- Checks: file_exists, file_changed, command, git_pushed
- Usage: `governance.VBR.Check(taskID, "file_exists", path)`

**ADL (Anti-Divergence Limit):**
- Prevents persona drift
- Loads baseline from SOUL.md
- Usage: `governance.ADL.CheckDrift(agentID, currentBehavior)`

**VFM (Value-For-Money):**
- Prevents wasteful spending
- Tracks: model, tokens, cost, quality
- Usage: `governance.VFM.TrackCost(agentID, model, inputTokens, outputTokens, costUSD)`
