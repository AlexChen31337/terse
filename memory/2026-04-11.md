## 00:59 — Claude Code Plugin Opus Pool Deadlock Fix
- **Bug:** `acquire()` in `WorkerPool` never spawns new workers when pool is empty — only queues waiters. If all opus workers die simultaneously (gateway timeout cascade + orphan cleanup), the pool deadlocks permanently for that model.
- **Root cause:** `forceDead()` kills workers via `onAbort` → SIGKILL. If all 4 workers per instance die with no pending `release()`, `acquire()` queues a promise that never resolves.
- **Fix applied:** Added emergency worker spawning in `acquire()` when pool is empty and no workers busy. Added debug logging. File: `/media/DATA/.openclaw/extensions/claude-code/src/server.ts`

## CCP1 Tool Racing Bug — Active Investigation (08:25–08:56)

### Problem
After gateway restart, all Claude Code plugin (CCP) worker pools show `initialized: 0`. The `claude -p` children spawn but never emit the `init` event to the parent Node.js process. This causes all tool handlers to be unavailable — sessions get tool schemas in prompts but no working tool execution.

### Root Cause (still investigating)
- Direct `node -e` test spawning `claude -p` works fine (init fires in ~2s)
- But the gateway's plugin-spawned workers never get `handleData` callback from stdout
- Workers have socket-based stdin/stdout (normal for Node 22) but no data flows through
- Both CCP instances (port 8091, 8092) affected identically
- Plugin code IS deployed (both `~/.openclaw/extensions` and `/media/DATA/.openclaw/extensions` are same files, same inode)

### Fix Attempted (commit c19cc0d)
- Added `waitForInit()` gate in `Worker.call()` — blocks until init or 30s timeout
- Added `isModelReady()` and `getPoolStats()` to PersistentPool
- Health endpoint now reports initialized count per model
- Disabled stale systemd shim services (`claude-code-shim.service`, `claude-code-shim-1.service`)
- **Result:** Fix is correct conceptually but doesn't help because workers never initialize at all
