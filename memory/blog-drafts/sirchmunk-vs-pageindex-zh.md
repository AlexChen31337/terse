# 索引还是实时搜索？AI Agent 知识检索的两条路

*2026年3月 | 本文基于项目文档分析，非实测数据*

---

## 先说一个你肯定遇到过的问题

你给 agent 接上了代码库。它能推理，能写代码，能调工具。然后你让它找一个函数——某个三层目录下的实现，或者分散在五个文件里的配置项。

它开始幻觉了。

不是因为模型不够聪明。是因为那段代码根本没进上下文。垃圾进，垃圾出。

这就是 AI agent 知识检索的核心问题：**怎么把对的内容，在对的时间，塞到模型眼前。**

两种流派，两种赌注，走的是完全不同的路。

---

## 第一条路：先建索引（PageIndex 路线）

主流做法。先花时间把你的语料库——代码、文档、知识库——处理成向量索引或图索引。嵌入算好，关系建好，分块存好。

查询来了？毫秒级返回。索引做了所有重活，LLM 只看精筛过的相关片段。

这条路什么时候好用：
- 语料库巨大（几百万 token）
- 内容相对稳定
- 查询频率高，能摊掉建索引的成本
- 需要可复现的确定性检索

但代价是真实存在的。**索引反映的是建索引那一刻的数据状态。** 文件改了，索引就过期了。如果你的代码库每天都在提交，你要么不停重建索引，要么拿着过期数据当宝。

---

## 第二条路：直接搜原文（Sirchmunk 路线）

Sirchmunk 是 ModelScope（阿里巴巴旗下开源 ML 平台）2026年3月发布的工具，走的是完全相反的路——**没有索引**。

查询来了，它直接搜你的原始文件。实时的。

机制挺有意思：先跑 **ripgrep 关键词级联搜索**——快速、确定性的 grep 风格过滤，找出候选文件。然后跑 **Monte Carlo 证据采样**：用 LLM 做多轮采样，对候选文件建立概率性的相关性判断，把关键词匹配和上下文推理结合起来。

两种模式：
- **FAST 模式**：2-5秒，2次 LLM 调用
- **DEEP 模式**：10-30秒，更多采样轮次，适合复杂多跳检索

代价是每次查询都要调 LLM。但有个大优点：数据永远是最新的。v0.0.6，昨天（2026年3月12日）刚发布。

---

## 直接对比

| 维度 | PageIndex（索引优先） | Sirchmunk（实时搜索） |
|---|---|---|
| **启动成本** | 高（先建索引） | 零 |
| **查询延迟** | 毫秒级 | 2-30秒 |
| **数据新鲜度** | 可能过期 | 永远最新 |
| **费用模型** | 一次性建索引，查询便宜 | 按查询付费（LLM 调用） |
| **适合规模** | 大型语料库 | 聚焦型代码库 |
| **基础设施** | 需要向量数据库 | 只需 ripgrep + LLM |
| **变更频繁时的准确性** | 索引过期则下降 | 始终准确 |

---

## 什么时候用哪个

**用 PageIndex 的场景：**
- 内容稳定的大型知识库（内部 wiki、API 文档、法律文件）
- 查询频率高，能摊掉建索引成本
- 对查询延迟敏感（需要毫秒级响应）
- 有能力维护定期重建索引的 pipeline

**用 Sirchmunk 的场景：**
- 代码库或文档在频繁变更（每天都有提交）
- 不想搭向量数据库，零基础设施启动
- 原型阶段，或者 agent 只需要搜索聚焦的小型仓库
- 数据过期是不可接受的风险（比如让 agent 调试实时代码）
- 查询量不大，按次付费 LLM 成本可控

**一个粗糙但实用的判断标准：** 如果你的数据变化速度比你实际重建索引的频率还快，Sirchmunk 在准确性上就赢了。如果数据稳定、查询频繁，PageIndex 在成本和延迟上占优。

对于在活跃开发仓库上跑的编码 agent 来说，Sirchmunk 的实时搜索模型很有吸引力——agent 看到的永远是当前的代码，不是上周二的快照。

---

## 这两条路要去哪

Sirchmunk 刚出来这个事本身就说明一个问题：业界开始质疑"索引优先"是不是真的万能默认选项。

随着 LLM 推理越来越快越来越便宜，这个账开始算不拢了。5秒但永远准确的搜索，可能比50毫秒但拿着过期数据的搜索更值钱。搜索原始文件的每次 LLM 推理成本会持续下降。

更长远的趋势大概是混合路线：用快速关键词过滤（ripgrep、BM25）缩小候选集，再用 LLM 推理做最终的实时检索。Sirchmunk 已经在这么干了——只是没有预建的语义索引。

索引优先不会消失。对大型稳定知识库，它依然是对的工具。但实时搜索是个合法的替代方案，尤其是那些本来就不能忍受数据过期的 agent 工作负载。

知道什么时候用哪个——这是现在的新技能点。
