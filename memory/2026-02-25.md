### Memory Architecture Deep Dive

**Question: How does OpenClaw native outperform tiered hot/warm/cold approach?**

Key accuracy comparison (honest numbers):
| Metric | Tiered tree (Python) | Native hybrid (OpenClaw) |
|--------|---------------------|--------------------------|
| Precision@5 | ~88–93% | ~78–85% |
| Recall@10 | ~82–88% | ~85–92% |
| P(fires at all) | ~60–70% | ~100% |
| **Effective accuracy** | **~58–65%** | **~83–90%** |

Effective accuracy = precision × P(fires). Native wins because it always fires.

**What native loses:**
- No hot/warm/cold tiers — flat index, temporal decay partially compensates
- Chunk boundary problem (512 tokens may split mid-decision)
- No LLM reasoning path for ambiguous queries
- Cold archive gets low temporal scores, may not surface

**Correct hybrid approach:**
- Native OpenClaw = always-on baseline (90% of cases)
- `memory_search` tool = on-demand deep recall (already in OpenClaw toolset, uses native index)
- MEMORY.md = always-loaded curated context (workspace injection)
- Build tiered plugin only if native effective accuracy drops below 80% in practice

**Do NOT build OpenClaw plugin for tiered memory yet** — evaluate 1 week first. Trigger to build: missing past decisions, contradicting known context.

### RSI Loop Architecture Decision

**Question: Should we build OpenClaw plugin for RSI loop?**

**Answer: No plugin needed. Fix the data pipeline instead.**

RSI has two components:
1. **Outcome logging** (per-task) — currently manual, real gap
2. **Analysis/deploy** (periodic) — cron is fine, already works

**Decision: Wire EvoClaw tool loop → RSI JSONL auto-logging**
- ~150 lines of Go in toolloop.go
- RSILogger interface (JSONL + Noop) + WithRSILogger functional option  
- DeriveQuality() from error rate (5 buckets), DeriveTaskType() from tool names
- Writes to `skills/rsi-loop/data/outcomes.jsonl` (same schema as Python observer.py)
- Graceful no-op if data dir doesn't exist
- source: "evoclaw", one aggregate record per Execute() call (not per tool call)

**Why NOT a full plugin:**
- Analysis/deploy doesn't need lifecycle hooks
- Auto-deploying proposals that modify SOUL.md needs oversight
- Proposals from past runs persist — RSI isn't broken without plugin, just less precise

**PBR pipeline started:** `feat/toolloop-rsi-autolog` — Builder running as of 18:03

**RSI JSONL schema (exact):**
```json
{"id":"a1b2c3d4","ts":"ISO8601","agent_id":"main","source":"evoclaw",
 "task_type":"code_generation","model":"...","success":true,"quality":4,
 "duration_ms":6800,"issues":[],"tags":["toolloop","parallel"],"notes":"2 tool calls, 1 parallel batch"}
```

**Quality derivation:** 0 errors→5, <20%→4, <50%→3, <80%→2, ≥80%→1
**Task type inference:** bash/exec→code_generation, read_file→file_ops, websearch→web_search, git_commit→code_review, edge_call→infrastructure_ops
