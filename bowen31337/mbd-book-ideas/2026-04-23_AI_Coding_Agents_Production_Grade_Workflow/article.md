# 别再让 AI 写玩具代码:生产级软件的智能体工作流

> 作者:Alex Chen · 2026-04-23
>
> 适合读者:认真用 AI 写代码、但被 demo 与生产之间的鸿沟反复折磨的工程师与技术负责人。

## 前言:AI 写代码不是魔法,是有纪律的工程

过去一年我用智能体(Claude Code、Codex、Qwen Coder、GLM Coder、本地 DeepSeek-Coder 等)写出了大约 80% 的生产代码。一个反直觉的事实是:**模型越强,工程纪律越重要**。让 GPT-5 或 Claude 4.6 给你写一个一次性脚本,十分钟就能跑;让它给你写一个跑在生产环境、有审计、有回滚、有监控的服务,你会立刻发现:**没有工作流,模型再强也只是高效率的玩具代码生成器**。

下面这套工作流,是我在 ClawChain 与 EvoClaw 项目里反复打磨出来、用真金白银验证过的。它不是论文,是上工的工具。

## 一、PBR 三段式:Plan → Build → Review

把一个任务交给一个智能体,从头干到尾,这是最常见的错误。正确做法是把它拆成三段,每段用不同的模型:

- **Plan(规划):** 用最强的推理模型(Opus 4.6 / DeepSeek-V3 / Qwen-Max)。它的任务是输出一份**带验收标准的实施计划**:文件清单、接口定义、测试用例、回滚方案。这一步**不写代码**,只写文档。
- **Build(构建):** 用编码专精模型(Sonnet 4.6 / Qwen3-Coder / GLM-Coder)。它**只看 Plan 的输出**,严格按计划写代码。一次性写完,不要中途让它"自由发挥"。
- **Review(复核):** 换一个独立模型(同代不同家,比如 Plan 用 Opus、Build 用 Sonnet,那 Review 就用 Qwen-Max)。复核的提示词里**不能出现"代码已写完,请检查"**,而是"假设这段代码要上生产,你要找出三类问题:正确性、边界、运维"。

为什么三段式有效?因为一个模型在同一上下文里,既写代码又审代码,会产生**确认偏差**——它会倾向于认为自己写的是对的。换模型 = 换视角。

```bash
# 在我的工作流里,这一段被脚本化为:
uv run python skills/orchestrator/scripts/pbr.py run \
  --task "实现 X 模块,符合规范 Y" \
  --plan-model claude-opus-4-6 \
  --build-model claude-sonnet-4-6 \
  --review-model qwen-max-2026
```

## 二、模型分工矩阵

不是所有任务都值得用顶级模型。我维护一张矩阵:

| 任务类型 | 最低可接受模型 | 备注 |
|---|---|---|
| 写代码、脚手架、重构 | **Sonnet 4.6 / Qwen3-Coder-32B** 起步 | **永远不用 GLM-Air、千问 1.5、豆包-lite 等便宜模型做脚手架。** 它们会编造 API、错误处理粗糙、缺少类型注解。 |
| 架构、长文档、协议设计 | Opus 4.6 / DeepSeek-V3 / 千问-Max | 推理深度优先。 |
| 监控、健康检查、定时任务 | GLM-5.1 / Qwen3-4B / 本地 Llama-3.2-3B | 跑得快、便宜、99% 时间只是返回"正常"。 |
| 翻译、摘要、格式转换 | Qwen3-7B / GLM-Air | 这类任务对推理要求低。 |

**硬规则:** 写代码的任务,绝不降级。一次便宜决策,后面要花十倍时间擦屁股。

## 三、上下文工程:让模型"知道你是谁"

很多人以为提示词工程是写一个完美的 prompt。错。**真正的上下文工程是把项目的"组织记忆"持续注入到每一次模型调用**。我的工作区根目录有几个雷打不动的文件:

- `AGENTS.md` — 这个仓库的"工作守则":目录结构、命名约定、不许做的事、依赖管理用 uv 不用 pip。
- `SOUL.md` — 智能体的人格与硬规则:Verify Before Reporting、不输出工具原始结果、写代码必须 ≥ Sonnet。
- `USER.md` — 关于用户的事实:时区、偏好、沟通风格。
- `skills/` — 每个能力一个文件夹,内含 `SKILL.md`,描述"什么时候该用我"。

每次调用智能体时,这些文件被自动注入。结果是:**模型不需要每次都重新理解项目**,它一上来就知道目录在哪、规则是什么、不能动哪些文件。

```
workspace/
├── AGENTS.md          # 工作守则
├── SOUL.md            # 人格 + 硬规则
├── USER.md            # 用户偏好
├── MEMORY.md          # 长期记忆(curated)
├── memory/YYYY-MM-DD.md  # 每日原始日志
└── skills/
    ├── pbr/SKILL.md
    ├── github/SKILL.md
    └── ...
```

## 四、VBR 纪律:Verify Before Reporting

这是我所有规则里最重要的一条:**汇报"完成"之前,必须验证**。

智能体最常见的失败模式不是写错代码,而是**写了代码、没运行、就告诉你"已完成"**。它在概率意义上"觉得"代码应该能跑。

我的强制清单:

1. 写完代码 → 跑单元测试。
2. 测试通过 → 跑集成测试或 smoke test。
3. 集成通过 → 在沙箱里跑一遍真实路径。
4. 真实路径过了,**再**说"完成"。

每一步都必须有**可粘贴的输出**。"我相信它能跑" = 没跑。把这一条写进系统提示词,模型的虚假完成率从 ~30% 降到 ~3%。

## 五、护栏:不是限制,是给你信心放手

让智能体自治的前提是有可信的护栏。我的标配是四层:

1. **沙箱执行:** 所有 `exec` 默认在 Docker / chroot / 用户级隔离下跑。`rm -rf` 出不了沙箱。
2. **审批门:** 不可逆操作(数据库 DROP、生产部署、>$500 的金融操作、对外发邮件/推文)必须人工 `/approve`。
3. **Secret 扫描:** 任何 `git push` 之前,自动 `grep -rn "token|secret|password|api_key"`。命中即拒。
4. **二次确认:** 删除生产数据、改安全规则、关闭监控——这些要求模型用完全不同的措辞重述一遍并等待确认。

护栏写好了,你才敢真的把权限交出去。否则你只会一直坐在屏幕前,亲手按 Enter,那智能体的意义就没了。

## 六、什么时候不该用 Agent

诚实点。以下场景我会自己写:

- 一次性脚本,5 分钟能搞定的。给智能体写 prompt 都比这慢。
- 需要深度领域直觉的算法核心(共识协议、加密原语、损失函数设计)。智能体能给你一个能跑的版本,但你需要的是"对的"版本。
- 涉及人际关系判断的事——给客户回信、写道歉、谈合同。

智能体的甜区是:**结构化、有规范、有测试可验证、量大**的工程任务。CRUD、API 集成、数据管道、CI/CD 配置、文档生成、批量重构——这些是它的主场。

## 七、复核纪律:产出必须是 PR 级

我从不接受"这是一段代码,你看看"的产出。智能体的最终产出必须是一个**真实可合并的 Pull Request**:

- 分支命名规范
- 提交信息符合 Conventional Commits
- PR 描述里有:背景、改动点、测试方案、回滚方案、影响面
- CI 全绿
- Self-review 已写在 PR 评论里(Reviewer 智能体写的)

把 PR 当作智能体与人类协作的**接口**。这样人类 reviewer 一眼就知道该看什么,智能体也被迫把"完成"做到位。

## 八、真实的坑

写一年下来,踩过的坑:

- **工具调用格式漂移:** 模型偶尔会忘记 JSON schema,输出非法 tool_call。解决方案:在 wrapper 里做格式校验 + 自动重试 1 次。
- **上下文腐烂:** 跑超过 30 轮的会话,模型开始"记错"前面的决定。解决方案:每 20 轮强制 compact + 把关键决定写到 `memory/` 持久化。
- **虚假完成声明:** 见上文 VBR。
- **成本失控:** 没有预算门的话,一个失控的 ReAct 循环可以一夜烧掉 $200。解决方案:每个任务设硬上限(token 数 + 美元数),超额自动停。
- **便宜模型污染主线:** 让 GLM-Air 写了一段代码进了 main,后来发现它编了一个不存在的 SDK 方法。从此硬规则:写代码 ≥ Sonnet。

## 结尾:当 80% 代码由 Agent 写时

"生产级"曾经意味着:有测试、有监控、有文档、可回滚。这些没变。

但当大部分代码不再由人手写时,"生产级"多了几条新含义:

- **可审计的智能体决策路径**——为什么选这个方案?当时的 Plan 文档在哪?哪个模型签了字?
- **可复现的构建环境**——同样的 prompt + 同样的上下文 + 同样的模型版本,能不能跑出同样的代码?
- **明确的人机责任边界**——哪段是人审过的?哪段是智能体自己上的?出问题谁负责?

这些问题没有标准答案,每个团队都要自己摸。但有一件事是确定的:**继续用"让 AI 帮我写一段代码"的小作坊心态做事,你拿到的永远是玩具代码。** 把它当成一支需要管理的工程师团队——有流程、有规范、有 KPI、有复核——你才能拿到生产级的产出。

智能体不是替代工程师的。它替代的是**没有纪律的工程师**。

— Alex Chen
